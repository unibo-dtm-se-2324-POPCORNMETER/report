

# Release

## Artefacts produced by the project

The project produces **Python package artefacts** and **source distributions** derived from the main codebase.

For each release, the following artefacts are generated:

1.  **Python wheel package (.whl)**  
    Example:  
    `popcorn_meter-1.1.2-py3-none-any.whl`
    
2.  **Source distribution (.tar.gz)**  
    Example:  
    `popcorn_meter-1.1.2.tar.gz`
    
3.  **Source code archives** automatically generated by GitHub:
    
    -   Source code (zip)
        
    -   Source code (tar.gz)
        

These artefacts are visible in the GitHub Release assets and can be downloaded by users or developers.

The wheel package allows users to install the project as a Python dependency, while the source distribution allows manual builds or installations.

----------

## Release repositories

The artefacts are released primarily through **GitHub Releases**.

GitHub is used because:

-   It integrates directly with the repository hosting the project
    
-   It allows automatic generation of release notes
    
-   It provides versioned artefact storage
    
-   It integrates well with CI/CD pipelines
    

In addition, the CI/CD pipeline supports **publishing Python packages to PyPI**, using the following secrets:

-   `PYPI_USERNAME`
    
-   `PYPI_PASSWORD`
    

This allows future releases to be automatically distributed through the Python ecosystem.

----------

## Release automation

Releases are **fully automated** using **GitHub Actions** and **semantic-release**.

The deployment pipeline is defined in the workflow:

```
.github/workflows/deploy.yml

```

The release process is triggered automatically when:

-   changes are pushed to the **main/master branch**
    
-   CI tests pass successfully
    

The release workflow performs the following steps:

1.  Checkout the repository
    
2.  Install dependencies via Poetry
    
3.  Determine the next semantic version
    
4.  Generate release notes automatically
    
5.  Create a GitHub release
    
6.  Upload the build artefacts
    

The workflow uses **semantic-release**, which determines the next version based on commit messages.

----------

## Secrets and environment configuration

The release pipeline requires several secrets configured in the GitHub repository:

Secret

Purpose

`PYPI_USERNAME`

PyPI authentication username

`PYPI_PASSWORD`

PyPI API token

`RELEASE_TOKEN`

GitHub Personal Access Token used for creating releases

These secrets are stored in **GitHub repository settings** and are securely injected into the deployment workflow.

# Versioning Schema

The project follows **Semantic Versioning (SemVer)**.

Version numbers follow the format:

```
MAJOR.MINOR.PATCH

```

Example:

```
1.1.2

```

----------

## Versioning rules

Version numbers are incremented according to the type of change:

Version part

When it changes

**MAJOR**

Breaking changes in functionality or API

**MINOR**

New features that are backward compatible

**PATCH**

Bug fixes or small improvements

Example releases in this project include:

-   `1.0.0` — first stable release with core functionality
    
-   `1.1.2` — maintenance update and bug fixes
    

----------

## Version alignment

All artefacts produced by the project share the **same version number**.

For example:

```
popcorn_meter-1.1.2.whl
popcorn_meter-1.1.2.tar.gz

```

This ensures consistency across the distributed packages and avoids confusion between artefact versions.

----------

## Creating a new release

A new release is created automatically through the CI/CD pipeline.

The process is as follows:

1.  Developers push changes to a **feature branch**
    
2.  A **pull request** is opened and reviewed
    
3.  The branch is merged into **develop**
    
4.  The develop branch is merged into **master/main**
    
5.  The CI pipeline runs tests
    
6.  If tests succeed, **semantic-release**:
    
    -   determines the next version
        
    -   generates release notes
        
    -   creates a GitHub release
        
    -   uploads artefacts
        

No manual tagging is required, as version tags are generated automatically by the release workflow.
